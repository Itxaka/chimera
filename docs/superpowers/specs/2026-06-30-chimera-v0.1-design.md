# Chimera — cloud-hypervisor desktop fleet manager (v0.1 design)

Date: 2026-06-30
Status: approved (brainstorming)

## Summary

Chimera is a Linux desktop application for managing a fleet of
[cloud-hypervisor](https://www.cloudhypervisor.org/) virtual machines — a
virt-manager-style tool targeting cloud-hypervisor instead of libvirt/QEMU.

This document specifies **v0.1**, the first shippable milestone. It proves the
full architecture end-to-end with the minimum feature set. Later milestones
(console, metrics, snapshots, hotplug, etc.) layer on top and are out of scope
here.

## Decisions (locked during brainstorming)

| Topic | Decision |
|-------|----------|
| Desktop stack | Tauri (Rust backend + web UI) |
| Frontend framework | Svelte / SvelteKit |
| Process ownership | App owns VM lifecycle (no central daemon) |
| VM persistence | Detached VMs — survive app close, reconnect on relaunch |
| v0.1 scope | Core lifecycle + VM list (no console/metrics/snapshots yet) |
| Boot model | Firmware/UEFI boot from a bootable disk image only |
| Networking | tap + bridge (privileged, full L2) |
| Privilege model | polkit + dedicated privileged helper binary |

## v0.1 scope

In scope:

- Dashboard listing VMs with state and basic info (cpu/mem from `vm.info`).
- Create wizard: name, vcpus, memory, bootable disk image, firmware path,
  bridge name, one network interface.
- Lifecycle actions: start, stop (graceful → power-button → kill), pause,
  delete.
- Detached VM processes that survive app close.
- Reconnect to running detached VMs on app relaunch.
- tap+bridge networking via a privileged helper, gated by polkit.

Explicitly NOT in v0.1 (roadmap):

- Serial/console access in-app.
- Live metrics / counters.
- Snapshots / restore.
- Device hotplug, cpu/mem resize.
- passt/pasta unprivileged networking.
- cloud-init / templates.
- Multi-host management.
- Direct kernel boot (`--kernel`/`--initramfs`); firmware/disk boot only.

## Architecture

```
┌─ Svelte UI ──────────────┐   dashboard · create wizard · VM detail
└──────────┬───────────────┘
           │ Tauri IPC (commands/events)
┌──────────▼ Rust core (in-app) ───────────────────────┐
│  vmm-client   → ch REST over unix socket (per VM)     │
│  supervisor   → spawn detached ch, pidfiles, reconnect│
│  store        → persist VM definition + runtime state │
│  net-client   → calls privileged helper via polkit    │
└──────────┬────────────────────────────────────────────┘
           │ D-Bus/pkexec (polkit)
┌──────────▼ chimera-netd (separate privileged binary) ─┐
│  ONLY: create tap · attach to bridge · teardown        │
└────────────────────────────────────────────────────────┘
```

Privilege isolation: the web UI and Rust core run as the unprivileged user.
Only `chimera-netd` performs network operations, gated by a polkit policy. The
app never runs privileged code in-process.

Each VM is one detached `cloud-hypervisor` process with its own
`--api-socket`. There is no central VM daemon — the app reconciles its store
against live processes on launch.

## Components and boundaries

Each unit has one purpose, a defined interface, and is independently testable.

### vmm-client
Thin Rust wrapper over the cloud-hypervisor HTTP API on a unix socket.
Endpoints used in v0.1: `vmm.ping`, `vm.create`, `vm.boot`, `vm.shutdown`,
`vm.power-button`, `vm.pause`, `vm.resume`, `vm.info`, `vm.delete`. Holds no
state. Tested against a mock unix-socket HTTP server.

### supervisor
Spawns `cloud-hypervisor` detached (own session/process group) so it survives
app close. Writes a pidfile and records the socket path. Probes liveness (pid +
socket), kills processes, and on app launch reconciles the store against live
processes. Tested with a fake process (e.g. a sleep binary) and a synthetic
store.

### store
Persists one file per VM under `~/.config/chimera/vms/<id>.toml`. Splits
**definition** (desired config) from **runtime** (volatile state) so a crash
never corrupts desired config. Tested for round-trip and schema-version
migration.

### net-client / chimera-netd
`net-client` (in-app) calls the `chimera-netd` privileged helper via
polkit/pkexec. `chimera-netd` does only: create tap, attach tap to a bridge,
return the tap name, and teardown. Installed with a polkit policy file.
`chimera-netd` command construction is unit-tested; a root-gated integration
test sits behind a feature flag (skipped in default CI).

## State model

Two records per VM, persisted separately:

- **Definition** (user intent, stable): id, name, vcpus, mem, disks[],
  net{bridge}, boot=firmware/disk, firmware path, created-at.
- **Runtime** (volatile): pid, api-socket path, tap name,
  status (`creating | running | paused | stopped | failed`), last-error.

Status is derived and confirmed: on launch never trust stored status alone —
probe pid + socket, then set the truth.

## Data flow

### Create VM
1. Wizard collects: name, vcpus, mem, disk image path, firmware path, bridge.
2. Core writes the definition to the store.
3. net-client → `chimera-netd`: create tap, attach to bridge → returns tap name
   (polkit auth).
4. supervisor spawns `cloud-hypervisor --api-socket …` detached.
5. vmm-client: `vm.create` (payload from definition, tap passed as net device)
   → `vm.boot`.
6. Runtime updated (pid, socket, tap, status=running) → UI event refreshes list.

### List / refresh
Core iterates the store → `vmm.ping` each socket → `vm.info` for state/cpu/mem
→ emits to UI. Polled on an interval.

### Stop
`vm.shutdown` (graceful) → fallback `vm.power-button` → fallback kill pid →
`chimera-netd` teardown tap → status=stopped.

### App relaunch (reconnect detached)
Read store → for each VM check pidfile alive + socket responds → reattach as
running, else mark stopped and clean orphan tap.

### Delete
Ensure stopped → teardown tap → `vm.delete` if live → remove store file. The
disk image is left in place unless the user opts to delete it.

## Error handling

- **ch crash**: pid gone or socket dead while status=running → mark `failed`,
  surface last log line, leave tap for cleanup sweep.
- **polkit denied / netd fail**: abort create, leave no orphan process, show
  the auth error.
- **socket timeout**: bounded retry on ping, then `failed`.
- **orphan tap**: teardown is idempotent; reconcile-on-launch sweeps taps with
  no live VM.
- **partial create**: each step is reversible. If `vm.boot` fails, kill the
  process and teardown the tap but keep the definition (status=failed) so the
  user can retry without losing config.

## Testing strategy

- **vmm-client**: mock unix-socket HTTP server; assert request payloads and
  response parsing.
- **supervisor**: fake process for spawn/detach/probe/kill; reconcile logic
  with a synthetic store.
- **store**: round-trip and schema-version migration.
- **chimera-netd**: pure command-construction unit tests; root-gated
  integration test behind a feature flag (default CI skips it).
- **frontend**: wizard validation and state-store component tests.
- **E2E** (`/dev/kvm` required): one happy-path create → boot → stop, gated to
  a labeled runner, not default CI.

## Roadmap after v0.1

Serial console (pty/vsock + xterm.js) → live metrics (`vm.counters`) →
snapshots/restore → device hotplug + cpu/mem resize → passt unprivileged
networking option → cloud-init/templates → multi-host.
